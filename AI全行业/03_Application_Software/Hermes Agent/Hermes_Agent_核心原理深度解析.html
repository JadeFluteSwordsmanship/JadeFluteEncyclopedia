1. 核心问题：Hermes 到底在解决什么？

在开始拆解 Hermes 的技术原理之前，我们先搞清楚它想解决什么问题。

传统 Agent 的四大痛点
  • 记忆断层：对话结束后，所有上下文消失，下次从零开始
  • 能力固化：无法从经验中学习，重复犯同样的错误
  • 上下文爆炸：对话越长，Token 消耗越多，性能越差
  • 部署复杂：难以在云端长期稳定运行（AutoGPT 的失败教训）

Hermes 的核心承诺是："会随使用而成长的智能体"——从"会话工具"向"持续工作者"演进。

关键洞察：人类不会每次对话都从零开始。你的助理记得你的习惯、偏好、过往决策。Hermes 试图让 AI Agent 也具备这种"持续性"。
2. OpenClaw 的实现原理：Markdown + 规则系统

要理解 Hermes 的创新，我们先看 OpenClaw 是怎么做的。

2.1 OpenClaw 的核心架构

OpenClaw 是目前最成熟、生态最大（13,000+ 技能）的开源 Agent 框架。它的设计哲学是：

OpenClaw 的"身份-记忆-技能-规则"四件套
  1. SOUL.md — 定义 Agent 的身份和性格
  2. MEMORY.md — 长期记忆（人工管理）
  3. memory/YYYY-MM-DD.md — 每日日志（自动生成）
  4. Skills/ — 技能库（Markdown 格式描述）
  5. AGENTS.md — 行为规则和边界
2.2 OpenClaw 的记忆系统原理

OpenClaw 三层记忆模型

第一层：会话上下文（Context Window）

当前对话的所有消息，存在 LLM 的上下文窗口中（临时）

第二层：Compaction（压缩）

当对话过长时，用小模型生成摘要，替换中间部分

第三层：Memory Files（持久化文件）

MEMORY.md（长期记忆）+ memory/YYYY-MM-DD.md（每日日志）

核心机制：
1. 每次对话开始时，Agent 自动读取 SOUL.md、USER.md、MEMORY.md、今日和昨日日志
2. 对话结束前，Agent 被要求"如果有重要内容，写入 memory/YYYY-MM-DD.md"
3. MEMORY.md 需要人工维护（从每日日志中提炼）
2.3 OpenClaw 的技能系统

OpenClaw 的技能是 Markdown 文件，描述"如何完成某个任务"。

示例：一个典型的 OpenClaw Skill

# Skill: 每日持仓更新

## 触发条件
用户提供券商截图或交易记录

## 执行步骤
1. 读取持仓文件 ~/Trading/config/portfolio.json
2. 调用 westock-data 获取最新行情
3. 计算手续费（佣金 + 过户费 + 印花税）
4. 更新持仓数量和成本价
5. 写入 portfolio.json 和 trades/DATE.json
6. Git 提交

## 注意事项
- 手续费："万0.85免五" = 无5元起步价
- Python 环境：~/.global_env/bin/python（禁用系统 Python）
OpenClaw 技能的局限：
1. 技能需要人工编写（Markdown 格式）
2. 不会自动生成——即使 Agent 重复做同样的事100遍，也不会自动沉淀为技能
3. MEMORY.md 需要人工维护——Agent 可以写每日日志，但长期记忆需要人工筛选整理
3. Hermes 的核心创新：四层记忆 + 闭环学习

Hermes 在 OpenClaw 的基础上，做了两大核心升级：

3.1 四层记忆架构（vs OpenClaw 的三层）
层级 OpenClaw Hermes Hermes 的创新
第一层 会话上下文（Context Window） 核心记忆（Core Memory） ✅ 强制筛选最重要的信息（如用户名、常用偏好），每次必加载
第二层 Compaction（压缩摘要） 对话存档（Conversation Archive） ✅ 向量化 + 全文搜索，按需检索而非全部加载
第三层 MEMORY.md（人工维护） 技能库（Skill Library） ✅ 自动生成技能（闭环学习，见下文）
第四层 ❌ 无 长期偏好（Long-term Preferences） ✅ 知识图谱，适合长期当个人助理
设计哲学的差异：
- OpenClaw：文件就是真相源，Agent 读写 Markdown 文件
- Hermes：分层存放不同粒度的记忆，按需取用而非全部塞进上下文
3.2 闭环学习机制（Closed-Loop Learning）

这是 Hermes 最核心、最差异化的设计。

什么是闭环学习？

简单来说：Agent 在执行任务时，如果发现"这个做法值得记下来"，会自动生成一个技能（Skill），下次遇到类似任务直接调用。

类比人类：你第一次做某件事可能很慢，但做过几次后，会形成"肌肉记忆"或"经验套路"，下次直接套用。

闭环学习完整流程
1. 执行任务
→
2. 判断是否值得记录
→
3. 触发条件筛选
→
4. 自动生成技能
→
5. 存入技能库
→
6. 下次直接调用
→
7. 根据实践"打补丁"优化
4. 闭环学习机制深度拆解
4.1 技能生成的触发条件
什么情况下会自动生成技能？
  • ✅ 调用工具超过 5次
  • ✅ 中途出错 + 自己修复成功
  • ✅ 用户手动纠正过
  • ✅ 发现了比上次更快的方法
4.2 技能的结构（推断）

虽然 Hermes 的技术报告未公开代码，但从描述可以推断技能的结构：

Hermes Skill 结构（推断的 JSON 格式）

4.3 增量优化机制（vs 全量重训）
方案 传统方案（如微调） Hermes 闭环学习
更新方式 重新训练模型 在现有技能上"打补丁"
成本 高（需要 GPU、训练数据） 低（只是更新 JSON 配置）
速度 慢（小时级别） 快（实时）
灵活性 低（需要重新部署） 高（即时生效）
关键洞察：Hermes 不是在"重新学习"，而是在"经验沉淀"。
经验沉淀 → 技能生成 → 技能复用 → 技能优化（增量更新）
4.4 "从失败中学习"的实现方式

虽然 Hermes 的文档没有明确使用"从失败中学习"这个表述，但从闭环学习机制可以推断：

错误修复流程
  1. 错误捕获：当任务执行出错时，记录失败上下文、使用的工具、参数、错误类型
  2. 自我修复：Agent 尝试不同的解决路径（如换个工具、调整参数）
  3. 修复成功 → 技能生成：将成功的修复路径固化为技能
  4. 技能优化：如果再次遇到类似错误，直接调用修复技能；如果发现更优方法，更新技能版本
维度 传统 Agent Hermes
错误处理 重试或报错退出 记录 + 修复 + 固化为技能
经验积累 ❌ 无 ✅ 错误修复路径成为可复用技能
下次遇到 重新处理（可能再次出错） 直接调用成功路径
5. 技术对比：OpenClaw vs Hermes
维度 OpenClaw Hermes Agent 优劣评价
核心定位 多平台接入的 Agent 工作入口 自进化的持续工作者 OpenClaw 更成熟，Hermes 更创新
记忆架构 三层（Context → Compaction → Files） 四层（Core → Archive → Skills → Preferences） Hermes 分层更细致
技能生成 ❌ 人工编写 Markdown ✅ 自动生成 + 增量优化 Hermes 核心优势
MEMORY.md 需要人工维护 自动管理 Core Memory Hermes 降低维护成本
生态规模 13,000+ 技能（绝对领先） 70+ 技能（差距巨大） OpenClaw 核心优势
部署方式 本地为主 + 多平台接入 云端 VPS + 多平台 各有优势（本地 vs 云端）
上下文压缩 简单摘要（Compaction） 四阶段智能压缩 + 增量更新 Hermes 工程更细致
子任务管理 单线程执行 3个子 Agent 并行 + 独立工作空间 Hermes 支持并行
GitHub 星数 35+ 万（成熟项目） 8-10 万（快速增长） OpenClaw 更成熟
核心差异总结：
- OpenClaw = "更好的工具" + 成熟生态
- Hermes = "会成长的系统" + 前沿理念

6.1 闭环学习的伪代码
闭环学习核心逻辑（Python 伪代码）

class ClosedLoopLearning:
    def __init__(self):
        self.skill_library = SkillLibrary()
        self.threshold_tool_calls = 5 # 触发条件：工具调用 > 5次

    def execute_task(self, task):
        # 1. 检查是否有可用技能
        applicable_skill = self.skill_library.match(task)
        if applicable_skill:
            return self.execute_with_skill(applicable_skill, task)

        # 2. 无技能，执行任务并记录
        result = self.agent.run(task)

        # 3. 判断是否值得生成技能
        if self.should_generate_skill(result):
            skill = self.generate_skill(result)
            self.skill_library.save(skill)
            print(f"✅ 新技能已生成：{skill['name']}")

        return result

    def should_generate_skill(self, result):
        """触发条件筛选"""
        return (
            result.tool_calls > self.threshold_tool_calls or
            result.self_repaired or
            result.user_corrected or
            result.found_better_method
        )

    def generate_skill(self, result):
        """生成技能结构"""
        return {
            "name": result.task_type,
            "tool_sequence": result.tools_used,
            "parameters": result.optimal_params,
            "success_rate": result.success_rate,
            "version": 1,
            "created_at": datetime.now()
        }
6.2 四层记忆的实现推断
四层记忆架构（Python 伪代码）

class FourLayerMemory:
    def __init__(self):
        # 第一层：核心记忆（必加载）
        self.core_memory = CoreMemory(max_size=1000) # Token 限制

        # 第二层：对话存档（按需检索）
        self.conversation_archive = VectorDB() # 向量数据库

        # 第三层：技能库
        self.skill_library = SkillLibrary()

        # 第四层：长期偏好
        self.long_term_preferences = KnowledgeGraph()

    def retrieve(self, query, context_window):
        """智能检索：组装最相关的记忆"""

        # 第一层：必加载
        context = self.core_memory.get_all() # 用户名、常用偏好

        # 第二层：语义搜索相关对话
        relevant_history = self.conversation_archive.search(query, top_k=5)

        # 第三层：匹配可用技能
        applicable_skills = self.skill_library.match(query)

        # 第四层：相关领域偏好
        preferences = self.long_term_preferences.get_relevant(query)

        # 智能组装，不超过 context_window
        return self.assemble(
            context, relevant_history,
            applicable_skills, preferences,
            max_tokens=context_window
        )
6.3 增量式上下文压缩
四阶段上下文压缩（Python 伪代码）

class ContextCompressor:
    def compress(self, conversation, previous_summary=None):
        """四阶段压缩"""

        # 阶段1：删除旧的工具调用结果（最占空间）
        conversation = self.remove_old_tool_results(conversation)

        # 阶段2：确定必须保留的内容
        must_keep = self.identify_critical_messages(conversation)

        # 阶段3：用小模型对中间对话做结构化摘要
        middle_messages = conversation[must_keep['start']:must_keep['end']]
        if previous_summary:
            # 增量更新：在上次摘要基础上更新
            summary = self.incremental_summarize(previous_summary, middle_messages)
        else:
            # 首次压缩：全新摘要
            summary = self.summarize(middle_messages)

        # 阶段4：修复摘要后可能破碎的工具调用配对
        compressed = self.fix_tool_call_pairs(
            must_keep['head'] + [summary] + must_keep['tail']
        )

        return compressed, summary # 返回压缩后的对话 + 摘要（供下次增量更新）
7. 结论：是真创新还是营销噱头？
7.1 真正的创新点
✅ Hermes 的技术贡献
  1. 四层记忆架构：分层清晰，按需取用，工程上优于 OpenClaw 的简单三层
  2. 闭环学习机制：理念先进，自动生成技能是真实的技术突破
  3. 增量式上下文压缩：在上次摘要基础上更新，而非每次重新总结
  4. 云端原生 + 多平台共享：设计更适合长期运行
7.2 尚待验证的部分
⚠️ 需要实际验证的问题：
1. 技能生成质量：自动生成的技能准确率如何？会不会生成错误的技能？
2. 长期运行稳定性：多次压缩后会不会出现信息失真？
3. 生态差距：70+ 技能 vs OpenClaw 的 13,000+，短期内难以弥补
4. 争议：EvoMap 指控"闭环学习"抄袭，若属实将动摇核心价值
7.3 对比 OpenClaw 的战略定位
维度 OpenClaw Hermes
产品形态 成熟的工具（像 IDE） 前沿的系统（像研究原型）
适用场景 立即生产可用 长期个人助理（尚需打磨）
技术路径 简单可靠（Markdown + 规则） 复杂创新（四层记忆 + 闭环学习）

核心价值 "现在就能用" "未来会更强"
7.4 最终评价

🎯 核心判断

Hermes 在"Agent 能否越用越强"这个方向上，提供了目前开源领域最清晰的技术实现路径。

虽然还不是完美的终态产品（生态不如 OpenClaw，技能质量待验证），但其架构设计具有真实的技术洞察和工程价值。

这不是单纯的营销噱头——闭环学习机制是真实存在的创新，只是需要时间证明其实际效果。

给开发者的建议：
- 如果需要立即可用的生产工具，选 OpenClaw
- 如果想探索"AI 助理会成长"的前沿方向，试试 Hermes
- 理想方案：两者结合——用 OpenClaw 的生态 + Hermes 的闭环学习理念
