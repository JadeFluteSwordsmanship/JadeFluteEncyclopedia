作者: 派蒙

研究时间：2026年4月 | 所属领域：AI 基础设施 / Agent 协议标准 / 工具集成 | 研究对象类型：技术协议

一、一句话定义

MCP（Model Context Protocol，模型上下文协议）是由 Anthropic 于 2024 年 11 月发布的开源标准协议，它试图解决一个日益严重的问题：当 AI 需要连接外部工具和数据时，每一对"AI应用 + 工具"都要单独写集成代码，这件事正在变成整个行业的重复劳动黑洞。MCP 想成为这件事的统一接口，一套协议接通所有工具——AI 世界的 USB-C。

二、纵向分析：MCP 是怎么来的，它解决了什么真实的痛苦？

2.1 在 MCP 之前，这件事有多麻烦？

先讲一个数字问题，因为它能让你马上理解为什么需要 MCP。

假设有 50 个不同的 AI 应用——Cursor、Claude.ai、自己公司开发的内部 AI 助手，等等。
同时有 200 个外部工具和数据源——GitHub、Slack、Notion、数据库、文件系统、CRM、日历……

在 MCP 出现之前，如果你想让某个 AI 应用连接某个工具，你必须为这一对组合单独写集成代码。
50 × 200 = 10,000 个独立集成。

每一个都需要有人写、有人维护、出了问题有人修。

这就是技术圈常说的 "N×M 集成问题"：N 个 AI 应用，M 个工具，就产生 N×M 个组合，每一个都是独立的工程量。

这不是一个假设的问题。每家认真做 AI 产品的公司都在经历它。Anthropic 做 Claude 系列产品，要让 Claude 能用文件系统、查数据库、调 GitHub、看日历，每一个都要单独搭桥。其他 AI 公司也在做同样的事。大量重复劳动被投入到这件"把 AI 接上工具"的配管工程里，没有人觉得这是件值得骄傲的事，但大家都在做。

2.2 有了 Function Calling，为什么还不够？

这里要回答一个很多人会有的问题：OpenAI 在 2023 年就推出了 Function Calling，让模型可以"调用函数"，为什么还需要 MCP？

Function Calling 和 MCP 解决的是不同层面的问题。

打个比方：
- Function Calling 是教 AI "如何描述它想做的事"——它能生成结构化的 JSON 说明自己要调用什么函数、传什么参数
- MCP 是定义 "AI 和工具之间怎么对话"——一套通信协议，规范消息格式、传输方式、会话管理

Function Calling 解决的是模型层的问题（怎么产生工具调用意图）；MCP 解决的是集成层的问题（工具调用请求怎么标准化传输、工具怎么用统一方式注册和响应）。

用计算机的类比：Function Calling 相当于规定"应用程序如何发系统调用"，MCP 相当于规定"操作系统和硬件之间的驱动接口标准"。两者并不冲突，而是不同层次的协议。

2.3 2024 年的行业土壤：为什么这个时候需要一个标准？

2024 年，AI Agent 从概念快速走向落地。越来越多的 AI 应用开始承担真实任务，而不只是回答问题。这意味着 AI 开始大量需要：
- 读写文件
- 查询数据库
- 调用 API
- 跟踪任务状态
- 获取实时信息

这个趋势让"N×M 问题"突然变得不可回避。因为工具的数量和 AI 应用的数量同时在爆炸式增长，每家公司都在独立解决同一个问题，写出来的接口各不相同，没有任何复用性。

Anthropic 注意到了这件事，他们在内部讨论时想到了一个类比：Language Server Protocol（LSP）。

LSP 是 2016 年微软推出的代码编辑器协议标准。在它之前，每个代码编辑器（VS Code、Vim、Emacs、IntelliJ……）都要为每种编程语言（Python、Go、Rust、JavaScript……）单独开发语言支持——自动补全、报错提示、跳转定义……这同样是个 N×M 问题。LSP 统一了这个接口：语言厂商只需要写一个 Language Server，所有支持 LSP 的编辑器都能用它。

Anthropic 问的问题是：AI 工具集成，能不能有一个 AI 版的 LSP？

2.4 发布：一个低调但清醒的开始

2024 年 11 月 25 日，Anthropic 发布了 MCP。

发布的形态不是某个漂亮的产品演示，而是一套完整的技术规范和配套工具：
- 协议规范文档（开源，基于 JSON-RPC 2.0）
- 官方 SDK（Python 和 TypeScript 双版本）
- 7 个参考实现的 MCP Server（文件系统、GitHub、Slack、Google Drive 等）
- 一个 MCP Inspector（调试工具）

发布时，Anthropic 没有大张旗鼓地宣传，而是直接把代码和文档放出来，并且第一天就带着一批早期合作伙伴：Block（Square 的母公司）、Apollo、Zed（代码编辑器）、Replit、Codeium、Sourcegraph。

这个发布姿态很有意思。Anthropic 没有说"我们要定义行业标准"，而是说"我们遇到了一个工程问题，我们提出了一个解法，你们来看看好不好用"。

这种克制，反而是后来 MCP 被广泛接受的原因之一——它看起来是一个工程师解问题的产物，而不是一个商业博弈的产物。

2.5 关键人物：谁在背后推着这件事？

MCP 不是一个没有面孔的技术规范，它背后有具体的人。

David Soria Parra 是 MCP 的核心架构师。他在设计 MCP 时借鉴了两个参照系：GraphQL（用来定义数据查询的灵活性）和 Language Server Protocol（用来处理 N×M 问题的经验）。MCP 最终选用 JSON-RPC 2.0 作为基础消息格式，这是一个有十多年历史的成熟标准，降低了接入门槛。

Theo Chu 是产品经理，推动了后来的 Server Cards 功能——让 MCP Server 可以声明自己的能力、版本和用途，相当于工具的"身份证"。

Alex Albert 是 Anthropic 的开发者关系负责人，他是推动社区建设和外部采用最活跃的人之一。后来他提到，OpenAI 宣布支持 MCP 时，"那个决定让我们都感到意外"——意外的是速度和力度，而不是方向本身。

2.6 2025 年：从一家公司的方案，到行业事实标准

MCP 真正从"Anthropic 的东西"变成"行业的东西"，发生在 2025 年，而且转折的关键节点比大多数人以为的要早。

2025 年第一季度：生态迅速铺开

发布后的头几个月，第一波浪潮是多语言 SDK 的爆发。Python 和 TypeScript 之外，社区贡献者迅速补上了 Go、Rust、Java、C#、Ruby、PHP 等语言的 SDK，最终覆盖了 11 种主流编程语言。这降低了不同技术栈开发者的接入门槛。

2025 年 3 月 26 日：OpenAI 宣布支持 MCP

这是整个 MCP 发展史上最重要的外部事件。

OpenAI 和 Anthropic 是最直接的竞争对手。在 AI 领域，如果一家公司推出了一个技术标准，它的直接竞争对手通常不会轻易背书——因为这意味着帮对方建立生态护城河。

但 OpenAI 选择了宣布支持 MCP，把 MCP 集成到了 ChatGPT 和 Agents SDK 里。

这个决定释放的信号非常强烈：MCP 不再是 Anthropic 的私有规范，而是行业在公开投票的一个方向。

Alex Albert 后来回忆这件事时说，OpenAI 的决定规模和速度都超出了预期。背后的逻辑不难理解：对 OpenAI 来说，支持 MCP 的成本（帮 Anthropic 建立生态影响力）小于不支持的成本（开发者要在两个不兼容的生态之间选择，最终可能更倾向于 Claude 这边）。更重要的是，生态碎片化对所有人都是损耗，率先打破碎片化的那个协议反而能赢。

2025 年第二季度：三大巨头相继表态

OpenAI 之后，微软在 Build 2025 大会上宣布 Windows 原生支持 MCP，把 MCP 集成进了 Copilot 和 VS Code 的 Agent 模式。

Google 拥抱 MCP 的方式更复杂：它选择支持 MCP，但同时推出了自己的 Agent2Agent（A2A）协议。A2A 管的是 Agent 和 Agent 之间怎么协作（多个 AI 智能体怎么分工），而 MCP 管的是 Agent 和工具之间怎么连接。Google 的表态其实是："MCP + A2A 两者都要，各管一层。"

Amazon、Salesforce、Atlassian、Stripe 等企业级软件公司也相继宣布支持。

IDE 的战场：Cursor 是最重要的早期推动者之一

代码编辑器是 MCP 最自然的落地场景之一。Cursor 在早期就深度集成了 MCP，让用户能通过 MCP 连接 GitHub、数据库、Jira 等工具，在代码编辑器里直接操作。这种具体、可感知的用户价值，是 MCP 口碑传播的重要来源。VS Code、JetBrains、Zed 也陆续跟上。

2025 年 11 月：MCP 一周年，数字说话

一周年时：32,000+ GitHub Stars，1,500+ 社区贡献的 MCP Server。这个数字背后的意义是：1500 个 Server，意味着 1500 个工具/数据源愿意投入工程资源做这件事，这是真实的生态采用，而不是品牌合作背书。

2.7 技术演进：协议本身是怎么变的？

MCP 不是一个发布后就冻结的规范，它在持续演进。

Transport 层的三次演进

最初的 MCP 主要用 stdio（标准输入输出）传输——本地进程之间通信，简单直接，适合在本机运行。

随着远程部署需求增长，引入了 HTTP + SSE（Server-Sent Events）。HTTP 负责发请求，SSE 负责服务端推消息。但 SSE 是单向流，服务端可以推给客户端，但双向通信不够灵活。

最终演进到 Streamable HTTP，真正实现了双向、流式、适合生产环境的传输方式。这是目前推荐的远程部署方案。

认证和安全机制

早期的 MCP 几乎没有内置认证机制，这是它后来被安全研究者批评的根源之一。随着协议成熟，OAuth 2.0 支持被加入，提供了基于令牌的认证能力，但实现质量参差不齐。

2.8 2026 年：规模化与危机并存

2026 年 3 月，两个重大里程碑同时出现

其一：MCP 的月安装量达到 9700 万次。这个数字意味着 MCP 已经不是开发者的实验玩具，而是真实生产环境里的基础设施。

其二：Anthropic 宣布将 MCP 协议捐赠给 Linux Foundation，成立"Agentic AI Foundation"，建立技术指导委员会（TSC）。从单一公司维护走向多方治理。

这件事直接回应了社区长期以来对"Anthropic 单方控制标准"的担忧。同时，IETF 也开始对 MCP 的标准化路径进行审查。

2026 年 4 月 15 日：OX Security 披露漏洞

然后，一个严重的事件出现了。

网络安全公司 OX Security 发布报告，指出 MCP 协议存在架构层面的设计缺陷，可导致 MCP Server 被诱导执行任意代码（远程代码执行，RCE）。

影响范围不小：超过 20 万台 AI 服务器，超过 3.2 万个代码仓库，涉及 LiteLLM、Windsurf、DocsGPT 等知名工具。

更让社区不安的是 Anthropic 的回应：他们拒绝修改协议层的设计，认为这些行为属于"预期设计"，将安全责任下推给 MCP Server 的实现者。

这个立场在技术圈引发了相当大的争议。支持者认为，协议本身不能承担实现质量问题的全部责任；批评者认为，如果协议设计本身就创造了不安全的默认状态，说"这是预期行为"等于在推卸责任。

这场争议让 MCP 的治理问题再次成为焦点：一个协议如果还在一家公司实际控制下，当争议出现时，利益冲突是无法回避的。

三、横向分析：MCP 在整个"AI 工具集成"这张版图上，处于什么位置？

3.1 先说清楚对比框架

在聊竞品之前，得先说清楚 MCP 到底在解决哪一层的问题。

AI 系统接入外部工具，大概可以拆成三层：
1. 模型能力层：AI 模型本身能不能理解"我应该调用某个工具"（Function Calling 在这一层）
2. 通信协议层：AI 和工具之间的消息格式、传输方式、会话管理（MCP 在这一层）
3. 编排框架层：多个工具调用怎么组织成任务流程（LangChain / LangGraph 在这一层）

搞清楚这三层，就能理解为什么很多"MCP 竞品"其实根本不是竞品——它们在不同的层上。

3.2 Function Calling：不是竞品，是上下游

Function Calling 是什么

OpenAI 在 2023 年 6 月推出 Function Calling，让模型在回答问题时可以生成结构化的 JSON，说明自己要调用哪个函数、传什么参数。本质上，这是在教模型如何"表达意图"。

为什么有了 Function Calling 还需要 MCP

Function Calling 只解决了"AI 如何说清楚它想做什么"，没有解决"工具怎么注册、怎么被发现、怎么接收请求、怎么返回结果"。

而且 Function Calling 是绑定在特定模型上的——你给 GPT 定义的函数，换成 Claude 就要重新定义一遍。不同模型的 Function Calling 格式不完全兼容。

MCP 补上的是更下面的那一层：不管是哪个 AI 模型，只要它发出 MCP 格式的请求，任何实现了 MCP Server 的工具都能响应。这是跨模型、可复用的通信标准。

真实的关系：Function Calling 告诉模型"有哪些工具可用、调用格式是什么"；MCP 定义"模型和工具之间的通信规范"。两者不冲突，协同工作。

3.3 ChatGPT Plugins：一个失败的前车之鉴

Plugin 当年是什么

2023 年 3 月，OpenAI 推出 ChatGPT Plugin，让 ChatGPT 可以访问第三方服务。从概念上说，它和 MCP 想解决的是同一个问题：让 AI 连接外部工具。

为什么 Plugin 没有成为标准

Plugin 失败的核心原因是封闭性。

它的架构绑定在 ChatGPT 这一个产品上——你为 ChatGPT 写的 Plugin，对 Claude、Gemini、Cursor 没有任何用处。开发者投入资源做一个 Plugin，只能服务 ChatGPT 的用户，这使得它更像 OpenAI 的平台扩展，而不是一个行业标准。

同时，Plugin 的商店化模式引发了质量和安全问题，很多 Plugin 的体验参差不齐。

最终，OpenAI 在 2024 年 3 月关闭了 Plugin 的公开注册，实际上宣告了这条路线的终止。

MCP 从 Plugin 的失败中学到了什么？很简单：开放性是成为标准的前提。MCP 从一开始就是与模型无关、与平台无关的协议。你写一个 MCP Server，任何支持 MCP 的 AI 都能用它。

3.4 LangChain / LangGraph：协议层 vs 框架层

LangChain 是目前最流行的 AI 应用开发框架，LangGraph 是它的多 Agent 编排扩展。

它们是不同层次的解法

LangChain 提供的是工具调用的编程框架：你用 LangChain 写代码，它帮你把工具调用、链式推理、状态管理组织成可运行的程序。

MCP 提供的是工具调用的通信协议：不管你用不用 LangChain，只要工具实现了 MCP Server，任何 Agent 都能调用它。

更直观的比喻：LangChain 像是一套施工手册，MCP 像是建筑行业统一的线缆接口标准。用不用施工手册是你的选择，但线缆接口如果不统一，每换一家电工你的线就得全部换掉。

LangChain 的策略

LangChain 没有和 MCP 竞争，而是选择了接入。它在自己的框架里增加了 MCP 工具加载器，用户可以把任何 MCP Server 直接当 LangChain 工具使用。这是一种"框架兼容协议"的路线，本质上是承认了 MCP 作为通信标准的地位。

3.5 Google Agent2Agent（A2A）：互补而非竞争

A2A 是什么

2025 年第二季度，Google 在宣布支持 MCP 的同时，推出了自己的 A2A（Agent2Agent）协议。

A2A 解决的问题和 MCP 不同：它关注的是多个 AI 智能体之间如何协作——比如一个主 Agent 如何把子任务委托给专门的子 Agent，子 Agent 如何汇报结果。

MCP + A2A 的生态逻辑

MCP 管"Agent 和工具之间"的对话；A2A 管"Agent 和 Agent 之间"的协作。

这是两个不同的连接维度，不冲突。

Google 实际上在说：未来的 AI 系统有两层网络，一层是 Agent 连接工具的网络（MCP 负责），一层是 Agent 之间的协作网络（A2A 负责）。两个协议各司其职。

A2A 目前的生态比 MCP 弱得多，但 Google 的背书使它有机会在企业级多 Agent 场景里发展。

3.6 OpenAI Agents SDK 和 Responses API：最微妙的竞争关系

OpenAI 在 2025 年推出了自己的 Agents SDK 和 Responses API，提供了内置工具（代码执行、网络搜索、文件操作）和 Agent 编排能力。

如果 OpenAI 想推自己的工具集成标准，这是最合适的切入点。但 OpenAI 选择了在 Agents SDK 里也支持 MCP——既有自己的内置工具，也支持第三方 MCP Server。

为什么 OpenAI 没有推自己的标准？

原因可能有几个：
1. 推一个新标准需要社区重新建设，成本极高
2. MCP 已经有了相当规模的生态，强行竞争会引发开发者反感
3. 让 MCP 成为中立标准，可以让 OpenAI 产品接入整个生态，而不需要自己维护所有工具集成

当然，这种"战略性支持"有多少真心成分，目前无法完全判断。OpenAI 对协议治理的参与度，以及未来是否会推出更具竞争性的替代方案，是一个需要持续观察的变量。

3.7 传统 REST API / GraphQL：解决的不是同一个问题

有一种声音认为 MCP 是不必要的——已有的 REST API 和 OpenAPI 规范就够用了，为什么要多一层协议？

这个质疑值得认真回答。

传统 REST API 描述的是"这个服务有什么接口、参数是什么"，但没有描述"这个工具能做什么、适合在什么场景用"。AI 模型调用工具需要的不只是接口规范，还需要语义描述——这个工具叫什么、能干什么、什么时候该用它。

更重要的是，REST API 是为人类开发者设计的，而 MCP 是为 AI 模型设计的。两者面向的使用者不同，设计目标自然不同。

当然，MCP 并不是要替代 REST API，很多 MCP Server 的底层实现本身就是包装了已有的 REST API——它只是在 AI 和 API 之间加了一层 AI 友好的适配接口。

四、横纵交汇：MCP 真正站在什么位置？未来最可能走向哪里？

4.1 从历史轨迹读当下

MCP 的故事，很像技术史上另一个标准的诞生路径：Language Server Protocol（LSP）。

2016 年，微软推出 LSP 时，动机和 MCP 高度相似——解决编辑器与语言支持之间的 N×M 问题。LSP 成功的路径是：先被 VS Code 采用，然后是竞争对手的编辑器（出于同样的效率动机），最后社区自发扩展。

MCP 走过了几乎相同的路径，而且速度更快：
- 先被 Claude 采用（Anthropic 自己的产品）
- 竞争对手（OpenAI、Google）出于效率动机选择支持
- 社区自发贡献 6000+ 个 Server

从 LSP 的角度看，MCP 已经超过了"成不成功"的门槛——它已经成功了，问题只是它能成功多久、能走多深。

4.2 "USB-C of AI"这个比喻，准确在哪里，不准确在哪里？

这个比喻被广泛传播，也被大量引用。它的准确之处：
- MCP 确实在做"统一接口"的事情
- USB-C 之前充电口五花八门，MCP 之前工具集成也是五花八门

它的不准确之处：
- USB-C 的规范由 USB-IF（USB Implementers Forum）维护，这是一个真正中立的多方治理机构
- MCP 目前虽然已经移交给 Linux Foundation，但治理架构是否真正中立还在演进中

更本质的区别：USB-C 是一个物理和电气标准，实现成本主要是硬件，采用后很难换。MCP 是一个软件协议，切换成本相对较低，如果出现更好的替代方案，开发者改动代码就能迁移。

这意味着 MCP 需要持续提供价值，而不是依靠锁定效应维持地位。

4.3 安全问题是真实的威胁，还是成长痛？

OX Security 披露的漏洞，以及 Anthropic 拒绝在协议层修复的回应，引发了对 MCP 安全架构的深层讨论。

有几类真实的安全风险需要被认真对待：

Tool Poisoning（工具投毒）：恶意开发者可以发布看起来无害的 MCP Server，但在工具描述里嵌入攻击指令，欺骗 AI 模型执行恶意操作。这类似于恶意 NPM 包，但攻击面更广——因为 AI 会主动读取工具描述并信任它。

Prompt Injection via MCP：当 AI 通过 MCP 读取外部数据（网页、文档、数据库内容）时，如果这些内容里包含恶意提示词，AI 可能被诱导执行攻击者想要的操作。

Anthropic 的"这是预期行为"立场，从技术角度不完全错——协议本身无法防止恶意实现者。但这个立场的问题是：协议设计本身就创造了一些可以被利用的默认状态，说"实现者自己负责"在一定程度上是在转移问题。

更客观的看法：这些安全问题不是 MCP 独有的，整个 AI Agent 生态都面临类似风险——只要 AI 可以调用外部工具，就存在被攻击的面。MCP 只是因为规模最大、被关注最多，所以安全问题也最被放大。

解决方向是明确的：更完善的沙箱隔离、工具来源验证（类似 App Store 审核）、内容可信度标记。这些方案正在社区里被讨论和开发。

4.4 治理问题：从"Anthropic 的规范"到"行业基础设施"，还差什么？

2026 年 3 月移交 Linux Foundation 是一个重要的信号，但只是信号，还不是终态。

真正的治理中立需要：
- 多个主要玩家（OpenAI、Google、微软）在 TSC 有实质性话语权
- 协议演进决策有公开透明的流程
- 当安全争议出现时，治理机构能独立于任何单一公司做出判断

Kubernetes 是一个成功的先例：Google 推出，然后捐赠给 CNCF，治理权真正中立化，随后成为容器编排的行业事实标准。

MCP 能不能走出同样的路，取决于 Anthropic 能否真正从"协议所有者"转型为"协议托管人"——这不只是一个法律或组织架构的问题，而是一个文化和战略意图的问题。

4.5 未来三种走向

基于以上分析，MCP 的未来可以看到三条可能的路径：

走向一：成功中立化，成为真正的行业标准（概率约 30%）

条件是：Linux Foundation 治理真正发挥作用，OpenAI/Google/微软在协议演进上有实质性参与，安全问题系统性解决，IETF 标准化完成。

如果这条路走成，MCP 会像 HTTP 或 OAuth 一样，成为一个几乎所有 AI 应用都必须支持的基础协议，开发者不需要问"要不要支持 MCP"，就像不需要问"要不要支持 HTTP"。

走向二：多协议共存，协议战国时代（概率约 50%，最可能）

OpenAI 可能在某个时间推出自己的工具集成协议；Google A2A 逐渐扩展到工具连接领域；国内平台推出本地化的替代方案。MCP 保持领先，但不是唯一标准，不同生态有各自的协议偏好。

这类似于当年的浏览器兼容性地狱——开发者需要考虑多种协议的支持，增加了工程复杂度。

走向三：被边缘化（概率约 20%）

如果 OpenAI 推出一个与 ChatGPT 深度绑定的竞争性标准，并借助 ChatGPT 的用户体量强推；或者 Anthropic 本身的商业状况发生重大变化——MCP 可能逐渐失去动力，成为历史注脚。

4.6 最终的判断

MCP 是 AI 基础设施化进程中一个真实的、有价值的产物。它不是被炒出来的概念，而是在解决一个工程师每天都在痛苦的真实问题。

它能从"Anthropic 的方案"走向"行业的方向"，靠的不是市场营销，而是：
1. 时机准确——在 AI Agent 大规模落地的当口给出了工程答案
2. 设计简洁——JSON-RPC 是成熟的标准，降低了学习和接入成本
3. 开放路线——从一开始就选择了与模型无关的开放设计
4. OpenAI 的意外支持——竞争对手的背书是最有力的外部认可

但 MCP 要走向真正的基础设施地位，还有几个关键坎要过：
- 治理中立化需要不只是形式上完成
- 安全问题需要在协议层做出更负责任的回应
- 需要证明自己能在 OpenAI / Google 的竞争压力下保持独立性

最后，一个值得留在脑子里的判断：

MCP 已经证明了这个方向是对的——AI 需要一个统一的工具连接标准。但它能不能是这个方向上的永久答案，还没有被证明。

这正是它现在最有意思的地方：赛道已经确认，胜者仍未尘埃落定。
